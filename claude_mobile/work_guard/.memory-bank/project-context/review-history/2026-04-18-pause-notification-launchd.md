# 2026-04-18 — Пауза, уведомления, LaunchAgent

## Симптомы
- Пауза визуально не включалась сразу; казалось, что срабатывает только после выхода/перезапуска.
- Непонятно, как корректно запускать (zsh без `conda init`, `conda activate` падает).
- `kill $(pgrep -f work_guard.py)` «не останавливает» приложение.

## Причины (подтверждены логами и кодом)
1. **`rumps.notification`** бросал исключение (нет `Info.plist` / `CFBundleIdentifier` у интерпретатора conda) — выполнение не доходило до обновления иконки до рефакторинга; плюс старый вызов `_update_icon()` без `paused=True` сбрасывал иконку в 🟢.
2. В **`com.workguard.plist`** было **`KeepAlive=true`** — после `kill` launchd немедленно перезапускал процесс.
3. Ожидаемое поведение: при незавершённой паузе в `config.json` приложение после старта должно показывать режим паузы (раньше иконка могла кратко быть 🟢 до первого тика).

## Изменения в репозитории
- `work_guard.py`: сначала `_update_icon(paused=True)`, уведомление в `try/except`; при старте, если `is_paused()`, сразу `_update_icon(paused=True)`; отладочная инструментация удалена после верификации.
- `com.workguard.plist`: **`KeepAlive=false`**.
- **`stop_workguard.sh`**: `launchctl bootout gui/$UID …`, PID-файл, `pkill` по шаблону.
- **`CLAUDE.md`**, **`setup.sh`**: запуск через `conda run`, остановка через `stop_workguard.sh`, пояснение про KeepAlive и plist.

## Заметки
- Настройки из отдельного процесса `settings_dialog.py` по-прежнему не подхватываются основным процессом без перезапуска (известный долг из обзора 2026-04-16/18).
